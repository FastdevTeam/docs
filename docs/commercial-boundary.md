# Commercial Boundary: Free Discovery vs Paid Discovery

Discovery can take meaningful time. To avoid accidental **free consulting** (and to set clear expectations with customers), we separate:

- **Pre-sales discovery (usually free):** qualify fit, align outcomes, and define a credible MVP + plan.
- **Delivery-grade discovery (paid):** validate unknowns deeply (workflows, integrations, NFRs, security) and produce reusable project assets.

This page explains the boundary and the practical triggers to move from free to paid.

---

## What is typically free (Pre-sales discovery)

**Purpose:** decide whether to proceed and enable a credible proposal.

**Typical scope**
- Phase 0 (Initial discovery intake)
- Phase 1 (Sales discovery call)
- Phase 2 **light** (workflow workshop to define 3 core journeys + MVP slice)

**Typical outputs (directional, not implementation-ready)**
- Problem statement, outcomes, and draft success metrics
- 3 core user journeys (happy path + obvious exceptions)
- MVP hypothesis (in-scope / out-of-scope) and major assumptions
- High-level solution outline (boxes/arrows) and ROM estimate range
- Risks/unknowns and a plan to validate them

**Key characteristic**
- Does **not** require customer system access or deep technical artifacts (data contracts, endpoint mapping, measurable NFR targets).

---

## What is typically paid (Delivery-grade discovery)

**Purpose:** reduce delivery risk by turning unknowns into decisions and delivery-ready artifacts.

**Paid discovery usually includes**
- Detailed workflows with exceptions and edge cases
- User stories with acceptance criteria at scale (beyond a small MVP list)
- Integration mapping with concrete specifics (auth flow, endpoints, data contracts)
- Data work (samples, profiling, migration mapping, identifiers, quality)
- Measurable NFR scenarios (performance/availability/reliability/security)
- Security/procurement evidence pack work (questionnaires, sub-processor/residency constraints)
- Custom **PoV / prototype** work beyond showing existing capabilities

**Key characteristic**
- Produces artifacts the customer can use even if they selected a different vendor.

---

## The conversion test (simple and operational)

Move to **paid discovery** when **any 2** of the below are true:

1) **Effort threshold:** more than ~2–4 hours beyond initial discovery sessions  
2) **Stakeholder breadth:** multiple customer stakeholder groups are required (IT/Security/Ops)  
3) **Customer dependency:** access to customer systems/data is needed (sandbox, API keys, data extracts)  
4) **Artifact depth:** outputs become implementation-grade (stories + acceptance criteria, integration specs, NFR targets)

---

## Mapping to the phased playbook

- **Phase 0 — Initial discovery intake:** Free
- **Phase 1 — Sales discovery:** Free  
- **Phase 2 — Workflow workshop:** Free (light) → Paid (deep)  
- **Phase 3 — Technical deep dive + NFR scenarios:** Paid (default)  
- **Phase 4 — Security/procurement track:** Paid when non-trivial  
- **Phase 5 — Prototype/PoV:** Paid (default)  
- **Phase 6 — Handover + backlog refinement:** Either (light in pre-sales; delivery-grade in paid engagement)

---

## Packaging options (recommended motions)

### Option A — Free discovery → Paid “Design Sprint” (fixed fee)
**Use when:** integrations/NFRs/security complexity is real or scope uncertainty is high.  
**Outputs:** validated workflows, integration map, measurable NFR scenarios, risks/mitigations, prototype plan, delivery-ready handover packet.

### Option B — Paid discovery credited on signature
**Use when:** customer resists paying upfront.  
**Mechanic:** discovery fee is deducted from implementation if signed within X days.

### Option C — Paid PoV package (timeboxed)
**Use when:** customer needs proof before commitment.  
**Outputs:** thin end-to-end slice with explicit success criteria; stubs/mocks allowed to timebox risk.

---

## Anti-patterns (avoid)
- Treating all phases as “free by default”
- Producing delivery-grade artifacts before commercial alignment
- Allowing “just one more workshop” without resetting expectations on scope and cost
